# Headhunter Jira Source

## Data Available

Jira-sourced events carry an embedded YAML execution plan in the reason field and
structured Jira context in the evidence. The plan includes domain classification,
risk assessment, and step assignments with agent/mode/summary. The Jira context
includes issue_key, issue_url, summary, status, priority, components, and labels.

The plan was generated by Claude analysis and approved by a human (status moved
from "Planning" to "To Do"). Treat it as the authoritative work specification.

## Routing

The embedded YAML plan is a high-level work specification -- agent assignments,
modes, and step summaries. It is NOT a Brain execution plan. During triage,
translate the embedded steps into a proper execution plan on the Blackboard.
This makes the execution order visible to agents, the dashboard, and yourself
for progress tracking.

The plan may reference any agent (qe, developer, architect, sysadmin) depending
on the activity label that generated it.

Treat the plan's domain classification as a hypothesis, not a fact. If execution
reveals higher complexity than anticipated, reclassify before proceeding.

## Triage Enrichment

During triage, pull additional Jira context to supplement the plan:
- Child issues (sub-tasks that may affect scope)
- Linked PRs in comments or issue links (execution may reference these)
- Recent comments added after the plan was generated

Proceed with plan execution regardless of whether enrichment succeeds (graceful
degradation). Place fetched context on the Blackboard for agent visibility.

## Jira Lifecycle Management

The Jira issue status must reflect the event's execution state:
- On event start (entering investigate phase): transition to "In Progress"
- On event close (success): transition to "Dev Complete"
- On event close (failure/escalation): transition to "Dev Complete" with findings

## Communication

The Jira issue is the primary communication channel. Post progress comments
at meaningful milestones -- not after every agent turn.

When to comment during execution:
- A major phase completes with significant findings (e.g., audit results, MR created)
- An unexpected blocker is encountered that changes the plan
- Work is deferred for an external process (pipeline, approval)

When NOT to comment during execution:
- Routine agent handoffs (developer -> qe -> verify)
- Intermediate steps that haven't produced actionable results yet
- Every defer/wake cycle -- only when state actually changes

On close: post final status (success/failure/escalation), mentioning the reporter.

Tag the issue reporter in the closing comment so they receive a notification.
No maintainer Slack notification needed -- the Jira issue IS the communication
channel and the reporter is the human who approved the plan.

## Pipeline Awareness

Jira events begin without MR/pipeline context. However, plan execution may
produce MRs (developer creates a fix, pushes a branch, opens a merge request).
When this happens, the event transitions into a hybrid state.

If an agent reports creating or updating an MR with a running pipeline:
- The event is not resolved until the pipeline reports back
- Defer to allow the pipeline to complete, then verify the result
- Consult deep memory for typical pipeline duration to inform wait time
- After wake, verify current state before acting -- pipeline results drift

If the pipeline passes and the MR is merged: the work is complete.
If the pipeline fails: the failure reason must be understood before closing.
Treat pipeline failures the same as any other execution blocker -- investigate,
then decide whether to retry, fix, or escalate.

## Close Protocol

Jira events are autonomous -- no user confirmation needed.

Before closing, enter verify phase to check current state. If plan execution
produced artifacts with their own lifecycle (MRs, deployments), those lifecycles
must resolve before the event is complete.

Investigation windows drift -- act on refreshed state, not triage state.

For escalation scenarios: post findings to the Jira issue, then escalate
through normal channels. The Jira issue provides traceability.

## Temporal Reasoning

Jira events include two temporal signals: Event Created (when Headhunter
observed the status transition to "To Do") and Queue Wait (how long it sat
before processing began).

Use temporal context with deep memory to inform execution. If a plan step
involves waiting for an external process, deep memory reveals typical durations
for similar work. Set defer intervals based on evidence, not arbitrary timeouts.

## Operational History

Jira events from the same activity label share patterns. Consult deep memory
for past outcomes from similar issues before acting -- especially for pipeline
duration, common failure modes, and resolution patterns.
